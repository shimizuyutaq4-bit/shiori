Overview

何を、なぜ作ったか

青空文庫のリーダーは数多いが、その多くは「読む」だけで「理解」を助けない。 そして名作が多すぎて、初心者ほど「どれを読めばいいか分からない」。この2つを解く。

課題

「読む」から「理解」へ

難語はその場で辞書やAIに聞け、本文を選択して質問できる。読書を止めずに、分からなさをその場で解消する。

アプローチ

体験を一つに束ねる

読む・調べる・ハイライト・感想・レコメンドを別々の機能でなく、 ひとつの読書体験として統合。UIは縦書きに馴染む「選択して操作」で貫いた。

到達点

131作品・全機能を単一HTMLで

収録131作品すべてに手書きあらすじと実測難易度を用意。依存ゼロ・ビルド不要で公開し、PWAとしてオフライン読書にも対応。

How it works

ブラウザの中で、本を組み立てる

サーバーを持たず、青空文庫の原文を取得して縦書きの本にする。各所に一つずつ設計判断がある。

01 — データ取得

Shift_JIS の原文を、ブラウザで組版

青空文庫のXHTMLをGitHubミラーから取得（CORS対応）。文字コードはブラウザ側でデコードし、 縦書きとルビはブラウザ標準機能で表現する。外字画像はURLを絶対化。

res = await fetch(rawUrl) // GitHub raw（CORS: *） buf = await res.arrayBuffer() text = new TextDecoder("shift_jis").decode(buf) // writing-mode: vertical-rl ＋ <ruby> はブラウザが描画

02 — 難易度は“実測”

勘ではなく、本文から数値化

全131作品の本文を取得して計測。旧仮名/旧字か・漢字率・一文の長さから難易度★を、 文字数から読了目安を算出する。だから「★3」より「★2・30分・はじめて向き」で選べる。

score = w_orth·(旧仮名/旧字) + w_kanji·漢字率 + w_len·一文長 diff = 段階化(score) → ★1..5 // 表示は● min = 文字数 / 450 // 読了目安（分）

Difficulty × Length

「難易度」と「分量」は別の軸

言葉の読みやすさ（横）と、読み終わる時間（縦）は一致しない。 『こころ』は言葉はやさしいが約6時間、『舞姫』は短いが旧字旧仮名で手強い。 二軸で持つことで「今日は30分で、やさしいものを」という選び方ができる。

やさしい・入門向き 手強い（旧字・文語）

AI as Design

手書きあらすじを、AIに渡す

「AIをつなぐ」だけなら誰でもできる。ここでは何を文脈として渡すかを設計した。 作品全文ではなく、自分で書いた要約を渡す——それだけで質もコストも変わる。

選択して質問

全文ではなく、この一文について

文学は「全体を要約して」より「この一節どういう意味？」の方が多い。本文をなぞって 現代語訳／語句の意味／表現の意図／見どころ、あるいは自由に質問できる。

重い処理は別スレッドへ

総ルビは Web Worker で

全漢字にふりがなを振る形態素解析（kuromoji.js）はメインスレッドを止める。 Web Workerに逃がし、スマホでもUIが固まらないようにした。ふりがなは「難しい漢字だけ」に限定。

Design Principles

設計思想

機能を並べることより、「読者が何をしたいか」に馴染む体験を優先した。

01

体験から逆算する

「今日は30分しかない」で本を選ぶ——その現実に合わせ、難易度と読了目安を実測して用意した。

02

最小構成で作り切る

React等を使わず単一HTMLで完結。依存ゼロ・ビルド不要。なぜその構成かを説明できる。

03

実データで検証する

難易度は131作品を実測し、作品URLもCSVから著作権フラグで抽出して一つずつ検証。勘に頼らない。

04

AIを“設計”として使う

ただ繋ぐのでなく、自作あらすじを文脈に渡してコストと品質を両立。渡すものを設計する。

05

プライバシーは端末内で

しおり・ハイライト・APIキーも外部に送らず、すべてブラウザ内に保存。バックアップの書き出しも用意。

06

引き算する

感情演出や環境音は作ったが、体験に効かないと判断して削除。足すだけでなく削る判断も設計の一部。

Learnings

得られた学び

  • AIを設計対象として扱う：文脈に何を渡すかで、質・コスト・速度が決まる。
  • ブラウザ標準を活かす：縦書き・ルビ・文字コードをフレームなしで解決。
  • 体感を数値へ翻訳する：「難しい/長い」を実測指標に落とす。
  • 重い処理の分離：Web Workerでモバイルのフリーズを解消。
